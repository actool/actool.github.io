Everest

Everest is an automatic conformance verification tool for testing reactive systems. Implementations can be checked against their respective specifications based on a more genral conformance relation using regular languagens. The classical IOCO relation is a particular case of the new conformance realtion and it is also provided by the tool.

Development

Everest is developed in Java and composed by four modules: View, Parser, Automaton Construction and Conformance Verification. Arrows indicate the data flow among the modules.

The Swing library is used in the View module to create the interface that deals with input and output data. The user can provide IUTs and specification models in Aldebaran format, the conformance veficiation mode, IOCO or language-based, and also the regular expressions that model (un)desirable behaviors. Verdicts of conformance testing and the associated details are produced by the tool.

The Parser module checks if the IUT and the specification models given in .aut files are valid. The Automaton construction module (A ver ainda quais operacoes e construcoes estao neste modulo) get underlying automaton from IOLTS/LTS model and also the automaton that recognize the regex received by view. The process of checking conformance between IUT and specification is performed by the Conformance verification module.

Aldebaran format (.aut)

An example of Aldebaran file format and its corresponding model. Input labels are tagged by '?' an output labels are tagged by '!'.

The header indicates the initial state, the number of transitions and the number of states, respectively.

Each line below the header is a transition, which is defined as a tuple (s,l,p), where 's' is source state, 'l' is the label of transition and 'p' is the target state

Interface

a ver devido as mudancas da interface.

Fields:

  • Conformance: conformance verification is based on language or IOCO;
  • Kind of models: the models of implementation and specification are IOLTS ou LTS;
  • Label: if model is IOLTS must inform how distinguish the inputs and output label, it's possible do automatically if .aut file contains label with tag '!' and '?', or inputs and outputs labels can be manually informed;
  • Input/output:this field will appear if is choose inform manually the input and output labels of IOLTS;
  • Implementation:inform the path where the implementation model is in .aut file
  • Model:inform the path where the specification model is in .aut file
  • Language D:can be filled if the conformance based on language is chosen, represents the desired behavior of implementation express in regex;
  • Language F:can be filled if the conformance based on language is chosen, represents the undesired behavior of implementation express in regex;
  • Verify conformance:clicking on the button will verify that the implementation complies with the specification;
  • Veredict:will inform the verdict whether or not the models conforms
  • Show test cases:will show the test cases that prove the nonconformity between an implementation and an implementation, in addition to the paths covered by the models when applying each of the test cases
